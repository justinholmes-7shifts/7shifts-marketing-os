You're working with two repos. They stack, so you don't pick one. Here's which is which, and when you use each.

Your repo

Your Workspace

Your own Claude Code setup: folders, your CLAUDE.md, your memory, your projects. Where you do the work. It never leaves your machine.

The main repo

The Marketing OS

The shared brain everyone works from: brand, ICP, segment sizing, positioning, the seven personas, voice, and skills. One copy, everyone pulls from it.

Which tool, and when

The repo is where your work lives. The tool is how you work on it.

SituationUse
A one-off you can finish in the conversation: a question, a quick draft, a gut checkChat
A real project, something ongoing that needs memory and our shared contextClaude Code, brain loaded

The tell is simple: if you're re-explaining context, or the work spans more than one sitting, you should be in Code.

Skip-permissions: start off, then turn it on

Start in Code without skip-permissions so you can watch what it does and approve each step. Once you trust it, you can launch Claude with permissions skipped and it moves a lot faster. From inside ~/7shifts-marketing-os, run:

claude --dangerously-skip-permissions

Your files here are markdown and docs, all git-backed, so a wrong move is just an undo. One caution: the flag applies to your whole Claude Code session, not only this repo, so Claude can also touch files outside the brain that are not git-backed. Use it when you're working in the brain, and launch plain claude otherwise.

Why two and not one

If we each kept our own copy of the brand guidelines, the day brand changes the palette, most of them would be quietly wrong. The shared repo means we update once and everyone's next piece of work is current. It also gets smarter every time someone pushes a learning back, which a pile of private folders never does.

Your workspace is your own station. The Marketing OS is the walk-in the whole kitchen shares. You don't each buy your own tomatoes, and when you figure out a better prep, you put it back in the walk-in for everyone.

To get going

These commands assume macOS. On Windows, run them inside WSL.

  1. Set up your own Claude Code workspace (see docs/set-up-your-workspace.md).
  2. Unzip the copy Justin shared into your home folder so you have ~/7shifts-marketing-os. (Have GitHub access and prefer git? cd ~ && git clone https://github.com/justinholmes-7shifts/7shifts-marketing-os.git.)
  3. Point your CLAUDE.md at the brain so it loads every session.
  4. Install the team skills so Claude Code loads them everywhere (safe to re-run; it skips any skill you already have):
    mkdir -p ~/.claude/skills && cd ~/7shifts-marketing-os && for s in skills/*/; do n=$(basename "$s"); [ -e ~/.claude/skills/"$n" ] && echo "skip $n" || ln -s "$PWD/$s" ~/.claude/skills/"$n"; done

Giving back

When a campaign teaches you something durable, tell Claude "share this learning." It checks the lesson is worth keeping, writes it into the brain, and pushes it so the whole team gets it on their next pull. (Justin sorts out repo access so your push lands.) Keep your drafts and work-in-progress in your own workspace. The brain holds the shared, finished, reusable stuff.

What's in the brain

  • context/ — company context, positioning, ICP and sizing, personas, voice, brand guidelines and assets, framework
  • skills/ — team skills, each a repeatable tool:
    • brand-designer — a senior 7shifts designer with a point of view. On-brand social posts, slides, web modules, and ad creative, and it pushes back on off-brand asks.
    • lifecycle-campaign-strategy — pressure-tests a campaign before you build it: the customer outcome, the one business metric, the audience, and the kill condition.
    • clear-communications-v2 — tightens any writing with the Pyramid Principle, answer first. Good for readouts and memos.
    • share-learning — gets a durable learning or campaign record into the brain and does the git push for you, so you never touch a command line.
    • skill-builder — how we build and review new skills to one standard, from Anthropic's skills guide. Use it before you make a new one.
    • session-log — keeps a per-session work log, one dated file per session, so you (or the next session) can pick up where you left off. Writes the file only, never commits.
  • campaigns/ — the shared campaign record
  • learnings.md — the team's accumulated lessons

Your workspace, the shared brain. Set yours up once, pull from the same source, give back as you learn.
